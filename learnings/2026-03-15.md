## 🧠 技术知识提取

### Docker 容器化部署
- **多阶段 Dockerfile** - 使用 Node.js 20-slim 基础镜像，分离构建和运行时环境
- **Docker Compose 服务编排** - 定义 app + nginx 双服务，配置健康检查、卷挂载、端口映射
- **Nginx 反向代理** - 将外部请求 (8080:80) 代理到应用容器 (3456)
- **健康检查配置** - 使用 curl 检查 API 端点，确保容器就绪
- **数据卷持久化** - 将数据库文件存储在 `/data` 并映射到宿主机 `./data/` 目录

### 数据库持久化设计
- **环境变量驱动配置** - `DATABASE_PATH` 允许动态指定数据库位置
- **目录自动创建** - 首次启动时自动创建数据库目录 (`fs.mkdirSync(dbDir, { recursive: true })`)
- **SQLite WAL 模式** - better-sqlite3 默认启用 Write-Ahead Logging，提升并发性能
- **容器路径管理** - 理解容器内路径 (`/app`, `/data`) 和宿主机路径 (`./data`) 的映射关系

### 端口与环境配置
- **容器端口 vs 主机端口** - 格式 `主机端口:容器端口`，容器内部监听端口由应用决定
- **端口冲突处理** - 主机端口 80 被占用时，需选择其他端口 (如 8080)
- **健康检查端点** - 必须指向容器内监听地址 (`localhost:3456`) 而非宿主机

## ⚠️ 踩坑记录

### 问题 1: 端口映射错误
- **现象**: 应用无法访问，端口连接失败
- **根因**: docker-compose.yml 使用 `3000:3000`，但应用实际监听 3456 端口
- **解决**: 改为 `3000:3456` 使主机 3000 映射容器 3456

### 问题 2: 健康检查失败
- **现象**: 容器持续重启，健康检查超时
- **根因 1**: Alpine 镜像缺少 wget 工具
- **根因 2**: 健康检查端点错误指向宿主机地址而非容器内 localhost
- **解决**: 改用 curl，端点改为 `localhost:3456/api/data`

### 问题 3: Nginx 端口冲突
- **现象**: nginx 容器无法启动，报端口 80 被占用
- **根因**: 宿主机已有 nginx 服务监听 80 端口
- **解决**: 修改映射为 `8080:80`，访问地址变更为 http://localhost:8080

### 问题 4: 数据持久化失败 (严重)
- **现象**: 点击同步按钮数据看似保存，但刷新页面后数据丢失
- **根因**: `server.js` 硬编码数据库路径为 `path.join(__dirname, 'm3u-player.db')`，创建在容器 `/app/` 内，未使用挂载的 `/data` 卷
- **解决**: 修改为读取 `DATABASE_PATH` 环境变量，并自动创建目录
- **验证**: 数据库文件出现在宿主 `./data/m3u-player.db`，容器重启后数据保留

## 💡 经验教训

### 核心原则
1. **容器路径隔离** - 容器内应用必须使用挂载卷路径存储持久化数据，容器销毁时 `/app` 文件会丢失
2. **环境变量优先** - 数据库路径、API 密钥等配置应通过环境变量注入，避免硬编码
3. **端到端验证** - 每次配置变更后必须完整测试 (build → start → refresh → validate persistence)
4. **首次运行鲁棒性** - 挂载卷首次存在时为空目录，应用需自动创建子目录或文件

### 调试流程
- **端口问题** → 确认容器内应用实际监听端口 (`netstat -tlnp`), 调整映射
- **健康检查** → 确保容器内有可用工具 (curl/wget), 使用容器内地址检查
- **数据丢失** → 进入容器 (`docker exec -it <id> sh`) 检查文件实际路径，对比挂载卷

### Docker 最佳实践
- 使用特定 Node.js 版本 (20-slim) 而非 latest
- 健康检查必须与应用就绪状态一致
- 开发环境与生产环境配置分离，但验证流程相同
- 所有文件写入操作应指向卷挂载点，除非明确是临时文件
