## 💡 经验教训

### 核心原则
1. **外部服务 TTL 必须自动化** - 任何基于时间戳的认证、会话、租约都应实现自动续期，手动维护是反模式
2. **配置来源单一事实** - agentId 等动态值应从服务端 `/agents` API 获取，而非硬编码或猜测
3. **守护进程模式** - 对于需要持续维护的外部集成，构建独立守护进程比嵌入主 agent 更解耦、更可靠
4. **自愈能力优先** - 系统应自动检测故障并恢复，而非依赖人工发现和干预
5. **日志完整性** - 守护进程必须输出详细日志，便于事后分析和调试

### 故障响应模式
- **检测 → 诊断 → 修复 → 验证** 的四步流程有效
- 文件 fallback 机制（写入日志）作为状态记录和通知手段
- 当沟通通道（sessions_send）阻塞时，文件系统作为可靠的后备

### 架构观察
- 主 agent 不加入看板的设计有效，避免自身状态污染团队视图
- Star Office 服务稳定性不足（多次意外停止），需要外部监控而非依赖其自愈

### 监控指标
- 服务可达性（HTTP 状态码、curl 返回值）
- Agent 认证状态（authStatus 字段）
- 最后推送时间（lastPushAt）
- 守护进程存活（PID、日志心跳）
